# **Стайлгайд по разработке и оформлению документации проекта Betula (https://github.com/bouncepaw/betula)**

## **1. Введение**

Настоящий стайлгайд устанавливает единые требования к подготовке, оформлению и поддержанию документации проекта **Betula** — социальной сети для сохранения ссылок и закладок.
Документ определяет правила написания, форматирования, использования разметки и терминологии, чтобы обеспечить единый стиль и удобство восприятия документации пользователями и разработчиками.

## **2. Назначение и область применения**

Стайлгайд предназначен для:

* авторов, пишущих и обновляющих документацию Betula;
* разработчиков, описывающих API, настройки и установку;
* технических редакторов и переводчиков.

**Область применения:**
Все виды документации, создаваемые в рамках проекта Betula, включая:

* пользовательскую документацию;
* технические описания и руководства;
* материалы для внутреннего использования (разделы wiki, README, справка).

## **3. Общие требования**

1. Документация создаётся в формате **Mycomarkup**.
2. Все тексты должны быть лаконичными, структурированными и легко читаемыми.
3. Оформление должно быть единообразным во всех разделах.
4. Все примеры, команды и фрагменты кода должны быть проверены на корректность.
5. Перед публикацией текст проходит проверку орфографии и пунктуации.
6. Ссылки должны быть актуальными и рабочими.
7. Изображения и примеры не должны содержать личных данных.

## **4. Стиль изложения**

1. **Язык:** русский или английский (в зависимости от раздела). Важно сохранять единообразие.
2. **Стиль:** нейтральный, технический, без эмоциональных выражений.
3. **Форма изложения:** от третьего лица, без личных местоимений.

   * Верно: *“Нажмите кнопку Save.”*
   * Неверно: *“Теперь вы можете нажать кнопку Save.”*
4. Использовать активный залог: *“Откройте файл конфигурации”*, а не *“Файл конфигурации должен быть открыт.”*
5. Избегать лишних слов, сложных оборотов, длинных предложений (до 20 слов).
6. При описании шагов использовать повелительное наклонение.
7. Термины и команды не переводить, если они закреплены в IT-среде (например, *token*, *feed*, *tag*).
8. Названия интерфейсных элементов (кнопки, поля, вкладки) выделяются **жирным шрифтом**.
9. Единицы измерения и форматы данных указываются в международной системе (например, MB, UTC, ISO 8601).

## **5. Правила оформления текста**

1. **Заголовки:**

   * Использовать уровни заголовков от `#` до `####` (1–4 уровень).
   * Заголовок начинается с прописной буквы, без точки на конце.
   * Пример:

     ```markdown
     # Установка
     ## Настройка окружения
     ```
2. **Абзацы:** отделяются одной пустой строкой.
3. **Выделение:**

   * **Жирный шрифт** — для кнопок, терминов, имён полей.
   * *Курсив* — для примечаний, уточнений, новых терминов.
   * `Моноширинный шрифт` — для кода, команд, конфигураций.
4. **Списки:**

   * Маркированные: `-` или `*`
   * Нумерованные: `1.`
   * Пример:

     ```markdown
     1. Откройте настройки.
     2. Введите адрес.
     3. Сохраните изменения.
     ```
5. **Цитаты:** использовать символ `>` в начале строки.
6. **Кодовые блоки:** выделяются тройными обратными кавычками с указанием языка:

   ```bash
   python main.py
   ```
7. **Ссылки:**

   * Внешние: `[Название](https://example.com)`
   * Внутренние: `[[/settings | Settings]]`
8. **Изображения:** добавлять с альтернативным текстом:

   ```markdown
   ![Пример интерфейса](https://example.com/image.png)
   ```
9. **Таблицы:** использовать для структурированных данных. Первая строка — заголовки столбцов.

## **6. Правила Mycomarkup / Markdown**

1. **Внутренние ссылки** — в двойных скобках:

   ```
   [[/about | О проекте]]
   ```
2. **Кодовые блоки** — оформлять через:

   ```myco
   ```

   #!/usr/bin/env python
   print("Hello, world!")

   ```
   ```
3. **Цитаты** — с символом `>` и пробелом после него.
4. **Изображения и галереи:**

   ```myco
   img {
      https://example.com/image.jpg { //Описание изображения// }
   }
   ```
5. **Таблицы:**
   Использовать синтаксис `table { }`, не допускать пустых ячеек.
6. **Заголовки:** `=`, `==`, `===` для уровней 1–3.
7. **Разделители:** использовать `----` между крупными блоками информации.
8. **Недопустимо:**

   * Смешивать Markdown и Mycomarkup в одном блоке.
   * Использовать несуществующие или внутренние теги других систем.

## **7. Шаблоны документов**

### **7.1. Шаблон раздела справки**

```
= Название раздела

Краткое описание цели раздела.

== Использование
Описание шагов, примеры команд или действий.

== Примечания
Дополнительная информация и ссылки.
```

### **7.2. Шаблон описания ошибки**

```
= Код ошибки (например, BET-114)

== Описание
Краткое объяснение причины ошибки.

== Возможные причины
- Неправильный адрес сервера;
- Отсутствие HTTPS-соединения.

== Решение
1. Проверьте настройки.
2. Измените протокол на HTTPS.
```

### **7.3. Шаблон инструкции**

````
= Установка Betula

== Требования
- Python 3.10+
- Git

== Установка
1. Клонируйте репозиторий:
   ```bash
   git clone https://example.com/betula.git
````

2. Установите зависимости:

   ```bash
   pip install -r requirements.txt
   ```

== Проверка
Запустите приложение:

```bash
python main.go
```

## **8. Глоссарий**

| Термин | Определение |
|--------|--------------|
| **Betula** | Приложение для сохранения и организации ссылок |
| **Mycomarkup** | Система разметки, используемая в Betula |
| **Tag (Тег)** | Ключевое слово для классификации записей |
| **RSS Feed** | Поток обновлений, на который можно подписаться |
| **Token** | Уникальный ключ аутентификации пользователя |
| **Bookmarklet** | Кнопка в браузере для быстрого сохранения ссылок |
| **Instance** | Отдельный сервер или установка Betula |

## **9. Примеры корректных и некорректных оформлений**

| Тип | Пример | Комментарий |
|-----|---------|-------------|
| Да | `[[/settings `&#124;` Settings]]` | Верное оформление внутренней ссылки |
| Нет | `[Settings](/settings)` | Неверный синтаксис для Mycomarkup |
| Да | **Нажмите кнопку Save** | Единый стиль выделения интерфейсных элементов |
| Нет | *Нажмите кнопку save* | Ошибка в регистре и выделении |
| Да | ```bash<br>python main.py``` | Правильное указание языка для кода |
| Нет | ```python main.py``` | Пропущены обратные кавычки |

## **10. Контроль и обновление стайлгайда**

1. Ответственность за актуальность стайлгайда несёт технический редактор проекта.  
2. Проверка и обновление выполняется не реже одного раза в **год** или при значительных изменениях в структуре документации.  
3. Все изменения фиксируются в журнале версий (Changelog).  
4. Предложения по улучшению принимаются от всех участников команды через систему контроля версий (pull request).
